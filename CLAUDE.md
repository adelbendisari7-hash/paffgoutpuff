# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

This is a single self-contained HTML file, [bali-trip.html](bali-trip.html), for a group trip planner (Bali & Java, 13 Aug–7 Sept, 5 friends). There is no build, lint, or test tooling, no package.json, and no git repository — the entire app is one `<style>` + markup + `<script>` block meant to be dropped into a host page/iframe. Edit the file directly; there's no compile step.

To preview changes, open the file directly in a browser (`file://.../bali-trip.html` works standalone now — see Persistence below) or embed it in a host environment that provides `window.storage`. There is no local dev server. After any change, sanity-check in a real browser (e.g. via the `agent-browser` skill): open the file, exercise the four tabs, and check the console for unexpected errors (the `host save failed, falling back to local storage` messages are expected when `window.storage` isn't present).

## Architecture

Everything lives in `bali-trip.html`, structured as three parts in document order: inline `<style>`, the DOM skeleton (`.app` > header/timeline + tabs + four `.panel` containers), and a single IIFE `<script>` that owns all state and rendering.

**Rendering model**: no framework — each panel has a `render*()` function that fully re-generates its `innerHTML` from in-memory state and re-attaches event listeners every call (`renderTimeline`, `renderItinerary`, `renderBudget`, `renderChecklist`, `renderActivities`). Any state mutation is followed by a full re-render of the owning panel rather than incremental DOM updates. Tabs are shown/hidden via the `hidden` attribute in `setupTabs()`, which also re-invokes the target panel's `render*()` on every click — so switching tabs always reflects the latest state (e.g. an activity confirmed in the Activités tab shows up immediately in Checklist/Itinéraire). Don't remove that re-render-on-switch call; it's load-bearing now that Checklist and Itinéraire both read from `tripData.activities`, which the Activités tab mutates.

**Data model**:
- `STOPS` (const, itinerary source of truth) — one entry per trip stop with `id`, `name`, `dates`, `code` (timeline stamp), `start`/`end` (ISO dates, used to highlight the current stop in the timeline and as the min/max bounds when confirming an activity's date — see below), `mapQuery` (free-text location string used to build Google Maps search links), and `accommodation` (`{mapsUrl, note, warning}` — the Airbnb/Booking.com location for that stop, rendered as a "Logement" block right under the stop's own Maps link; `mapsUrl` can be `null` when no usable listing link exists (e.g. a deleted Airbnb listing), in which case only `note` renders; `warning: true` colors `note` in clay/red for real problems — wrong area, no reviews, broken listing — vs. `warning: false` for neutral info like "exact address only in the Booking.com confirmation"). Also the city list for the Activités tab's city selector — every stop is a valid target. There is intentionally no static day-plan/narrative field on `STOPS` (it was removed) — the Itinéraire tab shows only the Maps link, the Logement block, confirmed activities, and notes, so that nothing "activity-shaped" appears there without having gone through the Activités tab's propose→confirm(+date) flow.
- `CHECKLIST_GROUPS` (const) — the original curated per-location activity list (`{t: name, p: price text}`), e.g. Ubud/Nusa Penida/Canggu/Uluwatu. It is **no longer rendered directly anywhere**; it exists solely as one-time seed data consumed by `seedActivitiesFromChecklist()` (see below). Edit it only if you want to change what gets seeded into new/reset trip data — it has no effect on an already-seeded `tripData`.
- `tripData` (mutable, persisted) — `{expenses: [], notes: {}, participants: [], activities: [], seeded: false}`. Expenses are `{id, desc, amountEUR, origAmount, origCurrency, cat, paidBy}` (`amountEUR` is always the canonical value used for totals/splits; `origAmount`/`origCurrency` preserve what the user actually typed, in EUR or IDR). `notes` is keyed by stop id → array of `{id, text, by}`. `participants` is a flat array of names used for the settle-up split (falls back to the set of people who've paid an expense if empty). `normalizeTripData()` backfills missing top-level keys after load/import so older saved payloads don't break newer code.
- `tripData.activities` — the single source of truth for both the Activités tab and, once confirmed, the Checklist and Itinéraire tabs. Each entry: `{id, stopId, name, mapsUrl, amountEUR, origAmount, origCurrency, priceLabel?, addedBy, status, date, checked, checkedBy}`. `status` is `'proposed'` or `'confirmed'`. `name`, `mapsUrl`, a price, and `addedBy` are required at creation (enforced in the add-activity handler, including `isValidMapsUrl` requiring `http(s)://`). `date` is `null` until confirmed — confirming requires picking a date via the inline `<input type="date">` on the proposed row, constrained to `[stop.start, stop.end]` (validated in JS, not just the HTML `min`/`max`, using `setCustomValidity` + `reportValidity()` so an invalid/empty date shows the browser's native validation bubble instead of silently confirming). `checked`/`checkedBy` are the "we actually did this" checkbox state, shared by the same activity object across the Checklist tab and the Itinéraire tab's per-stop "Activités confirmées" section — checking it in one place is reflected in the other on next render. `priceLabel`, when present (only on seeded activities), holds the original free-text price range (e.g. `"18-29€"`) and is preferred over `${amountEUR}€` for display, since `amountEUR` for seeded items is only a parsed representative number (see `parsePriceEUR`), not a precise figure. Anyone can confirm/check/delete any activity — no per-user ownership check, consistent with the rest of the app.
- `seedActivitiesFromChecklist()` — runs once per `tripData` (guarded by `tripData.seeded`), called from `init()` right after `loadAll()`. Converts every `CHECKLIST_GROUPS` item into a `status: 'proposed'` activity: price is parsed to a number via `parsePriceEUR` (returns `0` for strings starting with "gratuit", otherwise the first number found — this is a rough/representative value, not authoritative, which is why `priceLabel` preserves the original text), `mapsUrl` is a generated Google Maps *search* link (`buildMapsSearchUrl`, name + `stop.mapQuery`) rather than a real address, and `addedBy` is the literal string `'Programme initial'`. If you need to re-seed (e.g. after changing `CHECKLIST_GROUPS`), you have to reset `tripData.seeded` to `false` (and probably clear previously-seeded `activities` with `id` starting `seed-`) — there's no UI for this.
- `myName` (mutable, persisted separately) — the current user's display name, entered via the "Tu es :" banner on the Budget and Checklist tabs.
- `activityCity` (mutable, session-only, not persisted) — which stop's activities are currently shown on the Activités tab; defaults to `STOPS[0].id` and resets on reload, same pattern as `openStops`.
- `activitySearch` (mutable, session-only, not persisted) — the Activités tab's cross-city search query. When non-empty, `renderActivities()` ignores `activityCity` and searches `tripData.activities` by name across every stop instead (results show a city tag via `activityRowHTML(a, {showCity: true})`); the "Proposer une activité" form is hidden while searching, since a new activity needs one specific city. `renderActivities()` preserves focus/cursor position across its own re-renders on every keystroke (see the `searchWasFocused`/`searchCursor` dance at the top of the function) — don't remove that, or the search box becomes unusable (loses focus after every character since the whole panel's `innerHTML` is replaced on each `input` event).
- `NB_PEOPLE` (5), `AVANT_BALI_PER_PERSON` (350€), `A_BALI_PER_PERSON` (700€) — the recommended per-person budget for the Java leg (Jakarta→Banyuwangi) and the Bali leg (Ubud→Uluwatu) respectively. `BUDGET_PER_PERSON` and `TOTAL_BUDGET` are derived from these (`1050`/`5250`), not independently hardcoded — change the two per-phase constants, not `TOTAL_BUDGET` directly, to keep them consistent.
- `IDR_PER_EUR` (`let`, mutable) — starts at the hardcoded fallback `17000` and is overwritten by `refreshExchangeRate()` on every `init()` with a live EUR→IDR rate from the keyless Frankfurter API, cached in `localStorage` (`bali-trip-idr-rate` / `-ts`) for 12h to avoid refetching every load. `exchangeRateSource` (`'default' | 'cache' | 'live'`) drives the small "1€ ≈ X IDR · ..." note under the Budget tab's ledger so it's clear whether the shown rate is live. Network failure (offline, CORS, API down) is caught and silently falls back to the cached or hardcoded rate — never blocks rendering.

**Fixed identity (`nameSelectHTML`)**: the "Tu es :" banner, "Payé par" (expense form), and "Ajouté par" (activity form) all go through `nameSelectHTML(id, selected)`, which renders a `<select>` of `tripData.participants` once at least one exists, falling back to a free-text `<input>` before that (same element `id` either way, so calling code's `document.getElementById(id).value` reads work unchanged regardless of which one rendered). This exists specifically to stop "Adel" vs "adel" from being treated as two different people in `byPerson`/settlement — once participants exist, every name that flows into a calculation is guaranteed to be one of the exact canonical strings in `tripData.participants`, not free-typed text. If you add another "who is this" field anywhere, use `nameSelectHTML` for it too rather than a plain input.

**Countdown badge**: `renderCountdown()` (called once in `init()`, not re-rendered — the day doesn't change mid-session) computes days until `STOPS[0].start` and writes "J-N avant le départ" / "En plein voyage 🌴" / "Voyage terminé" into `#countdown-badge` in the header.

**Persistence**: three layers, tried in order, each optional and independently failure-tolerant:
1. **Supabase** (`SUPABASE_URL`/`SUPABASE_ANON_KEY`) — **live and configured** (project `fdhnqguqxdymjdesoqat`), this is the primary sync path. `loadAll()` reads the single shared row (`trip_state` table, `id = SUPABASE_TRIP_ID = 'bali-trip-2026'`) via `loadFromSupabase()`, and `subscribeSupabaseRealtime()` opens a Postgres-changes realtime subscription so every client's `tripData` updates live (full re-render of all four panels) when anyone else saves — no polling, no manual refresh needed. Verified end-to-end with two isolated browser sessions: session A wrote a participant, session B picked it up on load with no shared localStorage, and a second write from A appeared on B live with zero interaction. `saveTrip()` upserts to Supabase first. The `@supabase/supabase-js` UMD build is lazy-loaded from a CDN (`loadSupabaseSdk()`) only when `supabaseEnabled()` — if you ever blank out the two constants again, that network request stops entirely and behavior reverts to `window.storage`/`localStorage` only, unchanged. `SUPABASE_ANON_KEY` holds a **publishable** key (`sb_publishable_...`, Supabase's newer key format, functionally equivalent to the legacy JWT `anon` key) — safe to keep in client-side code, it's meant to be public; access control is entirely via the RLS policies below, not secrecy of the key. Table + policies (already applied to the live project; re-run only if the table is ever recreated):
   ```sql
   create table if not exists trip_state (
     id text primary key,
     data jsonb not null,
     updated_at timestamptz not null default now()
   );
   alter table trip_state enable row level security;
   create policy "public read" on trip_state for select using (true);
   create policy "public insert" on trip_state for insert with check (true);
   create policy "public update" on trip_state for update using (true);
   alter publication supabase_realtime add table trip_state;
   ```
   The RLS policies are wide open (anyone with the key can read/write) because the app itself has no auth — same trust model as the rest of the file, just now shared across devices instead of siloed per-browser.
2. **`window.storage`** — an externally-provided host API (not defined in this file), used only if the Supabase read comes back empty/fails. `trip-data` is stored shared (`true`); `my-name` is stored per-device (`false`).
3. **`localStorage`** (`bali-trip-data` / `bali-trip-my-name`) — the final fallback, and always written to on every save regardless of whether Supabase/`window.storage` succeeded, so the app keeps working fully offline/standalone in a plain browser tab.

**Budget calculations** (`computeBudget`, `computeSettlement` in the script's budget section): `computeBudget()` derives `total`, `remaining`, `byPerson`, and `byCategory` from `tripData.expenses` in a single pass. `computeSettlement(names, byPerson, total)` implements a simple greedy debt-simplification (equal split across `names`, then match largest creditor against largest debtor repeatedly) to produce a minimal-ish list of `{from, to, amount}` transactions for the "Qui doit combien à qui" card — it does not guarantee the theoretical minimum number of transactions, just a reasonable one. The "Suivi par personne" card in the Budget tab compares each participant's equal share of `total` (`total / participantNames.length`) against `BUDGET_PER_PERSON`, flagging over-budget in clay/red — this is a per-person view of the *overall* spend, not a per-phase breakdown, because individual expenses aren't tagged to a city/phase (the "Budget recommandé par phase" card above it is a static reference only, not computed from actual expenses).

**Adding to the itinerary**: extend `STOPS` directly for new stops — the rendering is data-driven and requires no other code changes. `STOPS[i].start`/`end` must stay in sync with the human-readable `dates` string (they're not derived from it) — the trip year (2026) is hardcoded there since `dates` doesn't include a year. There is no code-level "adding to the checklist/itinerary content" anymore: both come exclusively from confirmed `tripData.activities`, added at runtime through the Activités tab (or, for the initial curated set, via `CHECKLIST_GROUPS` + `seedActivitiesFromChecklist()` — see Data model above). Don't reintroduce a static per-stop narrative field without checking with the user first — it was deliberately removed so the Itinéraire tab shows only activities that actually went through confirmation.

**Styling**: dark theme via CSS custom properties on `:root` (`--bg`, `--gold`, `--green`, `--clay`, `--teal`, etc.), fonts loaded from Google Fonts (Fraunces for display/italic headings, IBM Plex Sans/Mono for body/data). Content is in French. No input sanitization/escaping is done anywhere (expense descriptions, notes, participant/activity names are injected into `innerHTML` as-is) — consistent throughout the file, acceptable for a private trip-planner used only by the 5 friends. The one exception is `activities[].mapsUrl`, which is written directly into an `<a href>` and is therefore format-checked (`isValidMapsUrl`, must start with `http(s)://`) to keep it from becoming a `javascript:` URI.
